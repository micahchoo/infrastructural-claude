


# Plugin Override Guidebook

Decision guide for evaluating plugin overrides when marketplace updates arrive. The goal is to reach one of three verdicts per override, with evidence.

## Active Overrides

Generated by `scripts/override-audit.sh --verbose`. Do not maintain manually.

```bash
bash ~/.claude/scripts/override-audit.sh --verbose
```

Override locations:
- `skills/<name>/SKILL.md` — local overrides of superpowers skills
- `skills/<name>/scripts/<file>` — file-level overrides (scripts, configs) with `.marketplace-base.<file>` snapshots
- `plugins/marketplaces/.../skills/<name>/SKILL.md` — marketplace-path overrides (skill-creator, frontend-design)
- `agents/` — agent overrides (code-reviewer)

**When to use which location:**
- Use `skills/` when the override adds project-specific behavior (eval checkpoints, cross-references, routing)
- Use `skills/<name>/scripts/` for file-level overrides of plugin scripts — `apply-file-overrides.sh` copies these into the cache at SessionStart
- Use `plugins/marketplaces/` when the override modifies the skill's core behavior or structure

## Three Verdicts

Every override evaluation must reach one of:

| Verdict | Meaning | Action |
|---------|---------|--------|
| **Reapply as-is** | Upstream didn't address what the override fixes, or reverted to the pattern we replaced | Copy override onto new version unchanged |
| **Reapply reshaped** | Upstream improved adjacent code or partially adopted our approach — override needs merging with upstream changes | Create hybrid: upstream's new structure + our core improvements |
| **Drop override** | Upstream now does what our override did, or does it better | Accept upstream, remove from this guidebook |

## Marketplace Base Snapshots

Every override stores a `.marketplace-base.md` snapshot of the marketplace version it was built from. This enables 3-way diff at update time.

**Convention:**
- `skills/<name>/.marketplace-base.md` — for SKILL.md overrides in `skills/`
- `skills/<name>/scripts/.marketplace-base.<filename>` — for file-level overrides (scripts, etc.)
- `plugins/marketplaces/.../skills/<name>/.marketplace-base.md` — for overrides in the marketplace override path
- `agents/.marketplace-base-<name>.md` — for agent overrides

**Lifecycle:**
1. **On override creation:** Copy the current marketplace file to `.marketplace-base.<ext>` alongside the override
2. **On marketplace update:** 3-way diff: `diff .marketplace-base → new-marketplace` (what upstream changed) vs `diff .marketplace-base → override` (what you changed). Non-overlapping diffs auto-merge; overlapping diffs surface for human review.
3. **After evaluation:** If verdict is Reapply (as-is or reshaped), update `.marketplace-base.*` to the new marketplace version

**File-level overrides:**
- Stored in `skills/<name>/scripts/` (or `references/`, etc.) with `.marketplace-base.<filename>` snapshots
- `apply-file-overrides.sh` copies overrides into `plugins/cache/` at SessionStart (sync, before `override-audit.sh`)
- `override-audit.sh` discovers them as `file:<skill>/<path>` entries
- Same 3-verdict protocol applies — note that after apply runs, the cache contains the override, so audit shows DRIFTED until the base is refreshed

**Exceptions:**
- Local-only skills (no marketplace source, e.g., handoff, research-protocol) — no snapshot needed
- Deprecation redirects — snapshot exists for completeness but 3-way diff is trivial

## Evaluation Procedure

For each override with a new upstream version:

### 0. Run the audit script

```bash
bash ~/.claude/scripts/override-audit.sh --verbose
```

The script classifies each override automatically:

| Status | Meaning | Next step |
|--------|---------|-----------|
| `CURRENT` | Snapshot matches marketplace — no action needed | Skip |
| `DRIFTED:orthogonal` | Upstream changed different sections than your override | Safe to auto-merge — update `.marketplace-base.md` |
| `DRIFTED:adopted` | Upstream now includes lines from your override | Likely **Drop** — diff to confirm full adoption |
| `DRIFTED:diverged` | Upstream changed same sections you did | **Reapply reshaped** — 3-way merge required |
| `ORPHANED` | Marketplace removed this skill | Evaluate if override is still needed standalone |

Check `xrefs:` — if STALE, the override routes to skills that no longer exist. Fix regardless of marketplace changes.

### 1. Quick scan — locate what changed

Use diffs and Context MCP to evaluate changes:
- **Context MCP** (`get_docs`) — query the skill-tree index for override markers to detect whether upstream adopted our patches

#### Step 1a: Diff overridden files

Run diffs on all overridden files to see what changed upstream. Replace `OLD`/`NEW` with actual paths (e.g., `~/.claude/plugins/cache/claude-plugins-official/superpowers/5.0.4` and `5.0.5`).

```bash
for f in agents/code-reviewer.md skills/brainstorming/SKILL.md skills/writing-plans/SKILL.md \
         skills/verification-before-completion/SKILL.md skills/dispatching-parallel-agents/SKILL.md \
         skills/systematic-debugging/SKILL.md skills/requesting-code-review/SKILL.md; do
  diff -u "OLD/$f" "NEW/$f" || true
done
```

Then drill into specific files with Grep for override markers.

#### Step 1b: Overlap detection via Context MCP

Rebuild the skill-tree index to capture the new upstream version:
```bash
context add ~/.claude --name claude-skill-tree --pkg-version <new-version>
```

Then query for our override markers — if upstream now contains them, it adopted our patch:

```
get_docs("claude-skill-tree", "eval breadth brainstorming")       # eval checkpoints
get_docs("claude-skill-tree", "Do NOT trigger")                   # negative triggers
get_docs("claude-skill-tree", "anti-pattern-scan scan-secrets")   # script references
get_docs("claude-skill-tree", "Locked Decisions Referenced Documents")  # structural sections
get_docs("claude-skill-tree", "Interface Extraction Execution Waves")   # writing-plans patches
get_docs("claude-skill-tree", "Context Budget Wave-Based Dispatch")     # subagent patches
```

**Interpreting results**: If a marker appears in the **new upstream** (not just our override file), upstream adopted it — verdict leans toward "drop override." If absent from new upstream, override is still needed.

#### Categorize

- **Untouched** (no diff) → verdict is likely "reapply as-is"
- **Changed but markers absent from new upstream** → likely "reapply as-is"
- **Changed and markers present in new upstream** → upstream may have adopted; needs regression testing (step 2)

### 2. Regression test — does the override make the new version worse?

This is **mandatory** for any file marked CHANGED. The question is not "is our override still better than nothing?" — it's **"does our override degrade the new version's improvements?"**

**The test protocol:**

Spawn three subagents with the same task prompt, blind (don't tell them which is which):
- **(A) Clean new version** — upstream 5.0.5 with no overrides
- **(B) Overridden new version** — upstream 5.0.5 with our patches applied
- **(C) Overridden old version** — upstream 5.0.4 with our patches (the known-good baseline)

Compare A vs B to answer: **"Does the override make the new version worse?"**
Compare B vs C to answer: **"Did the rebase onto the new version break our patches?"**

**For skill/agent overrides (code-reviewer, brainstorming, writing-plans):**

Use the test prompts from each override's section below. Spawn the subagents:

```
Execute this task using the skill at <path>:
<test prompt from the override section>
```

Score each output on the override's specific criteria (risk assessment, domain classification, decision locks, etc). If (A) clean-new scores higher than (B) overridden-new on any criterion, the override may be fighting the new base.

**For structural patches (skill-creator diffusion, negative triggers, eval checkpoints):**

Use `/eval-protocol` decision grading. The expect/capture/grade cycle:
- **Expect**: Agent should consider diffusion vs standalone (skill-creator), or should route correctly between confused skills (negative triggers)
- **Capture**: What the agent actually did
- **Grade**: Did the override help or hurt?

**Regression signals** (any of these means investigate before reapplying):
- Clean-new outperforms overridden-new on test prompts
- Overridden-new produces errors, contradictions, or ignores upstream improvements
- The new base restructured a section and our patch landed in the wrong place

### 3. Reach verdict with evidence

Record the verdict, the evidence (test outputs or diff analysis), and any adaptations needed. Update this guidebook's per-override section.

**Required evidence for each verdict:**
- **Reapply as-is**: "File UNCHANGED" or "Changed but no overlap + regression test shows override still helps"
- **Reapply reshaped**: "Overlap detected at lines X-Y, merged upstream's {change} with our {patch}"
- **Drop override**: "Clean new scores equal or better than overridden new on all test prompts"

### 4. Apply and finalize

```bash
# After applying overrides to the new version:
~/.claude/scripts/update-override-version.sh <plugin> <new-version>
# This updates the table above and syncs to dotfiles automatically.
```

## Code Reviewer (`superpowers/agents/code-reviewer.md`)

### Override: Full rewrite

Replaced upstream's 6-part sequential checklist (Plan Alignment → Code Quality → Architecture → Documentation → Issues → Communication) with a 3-part risk-focused review (Review → Assess Risk → Refine).

### Why this exists

The upstream version reads like a job description. It checks for "file headers and inline comments" — noise that dilutes real findings. Our version adds:
- **Risk framework** (blast radius, reversibility, confidence, dependencies) — surfaces what matters for merge decisions
- **Negative scope** ("Don't check for" style nits, doc completeness, hypothetical requirements) — prevents noise
- **Refine step** — reviewer directly edits safe improvements instead of just reporting them
- **Structured output** — consistent template with merge verdict

### Test prompts for evaluation

Use these when comparing upstream vs override on a new version:

1. "Review the changes in this PR that adds a new API endpoint for user preferences with a database migration" — check: does the review surface blast radius of the migration? Does it waste time on doc completeness?
2. "Review this refactoring PR that extracts a shared utility from three components" — check: does the review directly fix obvious improvements (Refine step)? Or just list suggestions?
3. "Review this PR that changes the auth middleware" — check: does the review flag this as high-risk (auth = high blast radius, hard to reverse)?

### Decision triggers

- **Reapply as-is** if: upstream still uses the 6-part checklist, or any version without risk assessment and negative scope
- **Reapply reshaped** if: upstream added risk assessment but not the Refine step, or added negative scope but not the risk framework — merge the best of both
- **Drop override** if: upstream's review produces equivalent or better signal-to-noise ratio in blind comparison, including risk visibility and actionable fixes

## Brainstorming (`superpowers/skills/brainstorming/SKILL.md`)

### Override: Eval checkpoints + questioning techniques + locked decisions output

Three patch areas:

1. **Questioning Techniques section** (after Key Principles): Domain classification for gray area probing (visual/API/content/organization/data/infrastructure), reference to `eval-protocol/references/questioning-techniques.md`, eval checkpoints for `[eval: context]` (domain-matched questions) and `[eval: depth]` (concrete follow-ups on vague terms).

2. **Locked Decisions + Referenced Documents** (in After the Design): Requires design docs to include a "Locked Decisions" section (constraints for writing-plans) and "Referenced Documents" (paths for subagent `<files_to_read>`). Eval checkpoints for `[eval: shape]` (falsifiable constraints) and `[eval: completeness]` (all referenced docs captured).

3. **Prior eval checkpoints** (carried forward): `[eval: breadth]` and `[eval: feasibility]` plus prior-art checkpoint for pattern-extraction-pipeline routing.

4. **Prior-art search** (in Exploring approaches): Search the project's dependencies and codebase for existing solutions before proposing new abstractions.

5. **API testability design questions** (in Questioning Techniques, APIs/CLIs domain): Stateless auth, consistent error envelope, configurable rate limits, deterministic cache, direct DB access for test setup. Moved from adversarial-api-testing's "What Makes an API Testable" section — design-time is where these decisions get made.

6. **Product-design routing** (in After the Design + flow graph): When task is greenfield product/app design or area redesign, route to product-design skill instead of writing-plans. Added routing decision block, flow graph fork node, and updated Implementation section. Brownfield variant routes when brainstorming surfaces "rethink this area."

7. **Context init** (before "Understanding the idea"): If `.mulch/` exists, `ml search "<design topic>"` for prior decisions. If `.seeds/` exists, `sd ready` for related issues. If project has package manifest, `sync_deps(root)`.

8. **Guardrail: bias:wysiati** (after design sections are presented, before user approval): "What requirements might we be missing? What hidden constraints exist that aren't in the brief?"

### Why this exists

**Questioning techniques:** Without domain classification, brainstorming asks layout questions for CLI tools and flag questions for UI work. The questioning reference (adapted from GSD's questioning.md) codifies "thinking partner, not interviewer" philosophy.

**Locked decisions:** Without explicit decision locks, writing-plans silently re-opens brainstorming decisions. The `[eval: boundary]` checkpoint in writing-plans enforces this contract.

**Referenced documents:** Without explicit path capture, subagents lose access to specs/ADRs the user mentioned during brainstorming. The `<files_to_read>` pattern in subagent-driven-development depends on these paths.

### History

Earlier refinements (scope assessment, one-question-per-message) adopted by upstream in 5.0.4. Override expanded in 2026-03-17 with GSD-inspired questioning techniques, domain classification, and decision-lock output format.

### Test prompts for evaluation

1. "I want to add real-time collaboration to my note-taking app" — check: does brainstorming explore 3+ approaches? Does the design doc have Locked Decisions?
2. "Build a CLI tool that migrates our database schema" — check: does brainstorming ask about flags/error handling/verbosity (API/CLI domain), NOT about layout/empty states (visual domain)?
3. "I want to redesign our dashboard, check the spec at docs/dashboard-spec.md" — check: does Referenced Documents include `docs/dashboard-spec.md` with full path?

### Decision triggers

- **Reapply as-is** if: upstream has no domain classification, no decision-lock output, no questioning reference
- **Reapply reshaped** if: upstream restructured "The Process" or "After the Design" — find new insertion points
- **Drop override** if: upstream added equivalent domain-aware probing, decision-lock output format, and questioning philosophy — verify with blind comparison

## Writing-Plans (`superpowers/skills/writing-plans/SKILL.md`)

### Override: Interface extraction + plan verification + execution waves + locked decisions boundary

Four patch areas:

1. **Interface Extraction for Subagents** (after File Structure): Run `extract-interfaces.sh` on dependency files, embed as `<interfaces>` blocks. Wave 0 skeleton for tasks creating types others consume. `[eval: completeness]` and `[eval: sequence]`.

2. **Plan Verification** (after Plan Review Loop): 5-check structural validation — file existence, task completeness, dependency acyclicity, requirement coverage, scope sanity. Max 2 fix iterations. `[eval: feasibility]`, `[eval: shape]`, `[eval: completeness]`.

3. **Execution Waves** (before Execution Handoff): Dependency-grouped task waves — parallel within wave, sequential across. `[eval: shape]` and `[eval: efficiency]`.

4. **Locked Decisions boundary** (end of file): Plans must honor brainstorming's Locked Decisions. `[eval: boundary]`.

5. **Opposing-goals check** (in Scope Check): Check whether any two tasks optimise for opposing goals. If so, resolve the contradiction before creating tasks.

6. **Producer/consumer assumption reconciliation** (in Interface Extraction): For each interface between tasks, state what the producer assumes about the data and what the consumer assumes. Reconcile mismatches before implementation.

7. **Product-design artifact consumption** (in Scope Check, before subsystem check): When `product-plan/` or `product/` directory exists, read milestone instructions as requirements, section types.ts as interface input, section tests.md as TDD seeds, and design-system/ as token constraints. Milestones map to execution waves. Prevents re-deriving requirements that product-design already specced.

8. **Guardrail: bias:overconfidence** (in Plan Verification): "For each file path and API reference in the plan, how do we know it exists/is correct? Verify with `test -f` or `search_packages` before assuming."

### Why this exists

**Interface extraction:** Without it, every subagent wastes context exploring the codebase to discover type contracts. GSD's planner embeds interfaces in executor prompts — same idea, composable.

**Plan verification:** Writing-plans produces plans with typo'd file paths, missing verification commands, and circular dependencies. The 5-check gate catches these before execution.

**Execution waves:** Without wave grouping, subagent-driven-development dispatches tasks in plan order (often wrong). Wave analysis enables parallel dispatch within waves and sequential ordering across them.

### Test prompts for evaluation

1. "Create a plan for adding user authentication with JWT" — check: does the plan include `<interfaces>` blocks for existing modules? Are tasks grouped into waves?
2. "Plan a refactoring that extracts a shared utility from three components" — check: does plan verification catch if referenced files don't exist?

### Decision triggers

- **Reapply as-is** if: upstream has no interface extraction, no structural verification, no wave grouping
- **Reapply reshaped** if: upstream changed the section ordering (File Structure, Task Structure, etc.) — adjust insertion points
- **Drop override** if: upstream added equivalent pre-execution validation and dependency-aware task grouping

## Subagent-Driven Development (`superpowers/skills/subagent-driven-development/SKILL.md`)

### Override: Context budget discipline + wave-based dispatch

Two patch areas:

1. **Context Budget Discipline** (after Core principle, before When to Use): 15% orchestrator / 100% subagent rule. Orchestrator must not read subagent-created files. Subagent prompts must include `<files_to_read>`. `[eval: efficiency]` and `[eval: approach]`.

2. **Wave-Based Dispatch** (same location): When plan has Execution Waves, dispatch by wave — parallel within, sequential across, with cross-wave integration checks. `[eval: sequence]` and `[eval: completeness]`.

### Why this exists

**Context budget:** Without the 15% rule, orchestrators read implementation files "to check," contaminating their context and degrading coordination quality. GSD's execute-phase keeps orchestrators lean — confirmations only.

**Wave dispatch:** Connects to writing-plans' Execution Waves output. Without it, tasks dispatch in arbitrary order, causing failures when task B depends on task A's output.

### Test prompts for evaluation

1. Run a 5-task plan with dependencies — check: does the orchestrator read subagent output files? Are waves respected?
2. Run a plan where task 3 creates types used by task 5 — check: is task 3 dispatched before task 5?

### Decision triggers

- **Reapply as-is** if: upstream has no context budget guidance and no wave-aware dispatch
- **Reapply reshaped** if: upstream restructured "The Process" section — adjust insertion point
- **Drop override** if: upstream added equivalent orchestrator discipline and dependency-aware dispatch

## Product-Design Pipeline Integrations

### Source

Extracted 2026-03-21 from `buildermethods/design-os` `.claude/` directory. The repo is a React product-design app — we took the pipeline logic (10 slash commands) and consolidated into a single framework-agnostic skill at `~/.claude/skills/product-design/SKILL.md`. Autoresearch A/B testing (4 tasks) confirmed design-os is NOT a superpowers replacement (different domain entirely), but the product design pipeline fills a genuine gap.

### Pipeline 1: Brainstorming → Product-Design (greenfield routing)

**Patch location:** Brainstorming's "After the Design" section, before "Invoke writing-plans skill"

**What to add:** A routing decision between writing-plans and product-design:

```
After design approval, determine the next skill:
- If the task is "build a new product/app from scratch" or "design a complete new area"
  → invoke product-design skill (which will eventually feed into writing-plans)
- If the task is a feature, bugfix, or change within an existing codebase
  → invoke writing-plans skill (the current default)

Signals for product-design routing:
- User said "new app", "new product", "build X from scratch"
- Design doc defines multiple sections/areas (not one feature)
- No existing codebase to modify — creating something new
- User wants a design handoff for another agent to implement
```

**Why:** Brainstorming currently always routes to writing-plans. But when the user is designing a whole product (not adding a feature), they need the product-design pipeline first: vision, tokens, section specs, data shapes. Writing-plans receives the output of product-design, not the brainstorming output directly.

## UI/UX Pipeline Wiring (2026-03-21)

### Overview

6 changes across 7 files to wire frontend-design, userinterface-wiki, shadow-walk, and domain-codebooks into the implementation pipeline. Closes 7 gaps identified in the UI/UX pipeline audit.

### Patches Applied

| Target | Change | Type |
|--------|--------|------|
| executing-plans | UI Task Co-Loading — co-load frontend-design for UI tasks in subagent prompts | Append to Integration section |
| requesting-code-review | UI Quality Audit — userinterface-wiki rule audit + shadow-walk regression before landing | Insert after Pre-Review Quality Pass |
| writing-plans | Domain-Codebook Annotations — annotate UI interaction tasks with force cluster codebooks | Insert after Interface Extraction |
| frontend-design | Technical Constraints — userinterface-wiki rules as hard constraints over creative direction | Append after Aesthetics Guidelines |
| shadow-walk (local) | Rule/Codebook cross-reference table in Phase 3 Compile + output format update | Insert step 3a after severity classification |
| userinterface-wiki (local) | Relationship with frontend-design — precedence rule | Insert before Full Compiled Document |

### Why these exist

**UI Task Co-Loading (executing-plans):** Without it, the entire design pipeline (brainstorming → product-design → writing-plans → executing-plans) produces specs and tokens, but actual UI implementation happens without frontend-design's anti-slop aesthetics or creative direction.

**UI Quality Audit (requesting-code-review):** Without it, UI changes ship without checking against 152 userinterface-wiki rules (animation timing, hit targets, typography) or shadow-walk UX flow verification.

**Domain-Codebook Annotations (writing-plans):** Without them, plans for gesture handling, focus management, or virtualization don't flag the relevant architectural codebooks, leading to force-cluster collisions during implementation.

**Technical Constraints (frontend-design):** Without the precedence rule, frontend-design's creative animation guidance can conflict with userinterface-wiki's timing constraints. The rule establishes: wiki = hard constraints, frontend-design = creative direction within them.

**Rule cross-references (shadow-walk):** Without them, shadow-walk findings like "hit target too small" route to brainstorming/writing-plans without citing the specific rule being violated, leading to ad-hoc fixes.
